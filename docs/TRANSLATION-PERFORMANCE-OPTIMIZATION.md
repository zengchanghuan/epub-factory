# AI 翻译速度瓶颈归因与优化方案

## 背景

大书翻译经常出现耗时过长、少量段落反复失败、最终无法交付的问题。典型错误包括：

- `untranslated response`
- `html_tag_mismatch`
- `Failed to parse LLM JSON`
- 上游请求超时或不稳定

这类任务通常包含上千到数千个段落，单本书总耗时主要由模型请求数量、模型响应时间、失败重试次数和后端编排策略共同决定。

## 初步判断

当前主要瓶颈大概率不是服务器 CPU 或内存，而是：

1. 大模型响应速度和稳定性
2. 重试放大效应
3. 后端任务编排粒度

服务器配置仍然需要观测，但不应作为第一优先级盲目升级。

## 三类瓶颈判断

### 1. 服务器配置

服务器资源主要影响：

- EPUB 解包、清洗、打包
- SQLite 或持久化写入
- 多用户任务并发
- worker 排队能力

如果出现以下现象，才优先考虑扩容：

- CPU 长期高于 90%
- 内存不足或发生 swap
- 磁盘 IO 持续过高
- 多任务并发时队列明显堆积

如果单本书主要时间消耗在模型请求等待上，升级 CPU 或内存收益有限。

### 2. 大模型响应速度和稳定性

这是当前最高嫌疑瓶颈。原因：

- 翻译任务需要大量 LLM 调用。
- 单次响应慢会直接拉长总耗时。
- 模型返回未翻译、HTML 标签损坏、JSON 格式异常时会触发补译。
- 少数坏段落反复重试，会放大整本书耗时。

重点观测指标：

- LLM 总耗时
- API 调用次数
- 平均/P95/P99 延迟
- 重试次数
- 单段补译次数
- 失败类别分布

### 3. 后端代码和任务编排

这里不是简单的代码质量问题，而是编排策略是否足够适配大书：

- 批大小是否合理
- 章节并发是否合理
- 请求并发是否能动态调整
- 失败段落是否拖住整本书
- 是否支持断点续跑
- 是否能只重翻失败段落
- 是否能把慢段隔离到补译队列

后端优化能显著降低模型抖动带来的放大效应。

## 第一阶段：做耗时归因面板

先不要盲目调参。每个任务需要输出以下结构化指标：

- 总耗时
- 预处理耗时
- Manifest/切段耗时
- 术语表耗时
- 书名/元数据翻译耗时
- 正文翻译耗时
- Reduce/打包耗时
- EpubCheck 耗时
- 总 chunk 数
- API 调用数
- 平均模型延迟
- P95/P99 chunk 延迟
- 重试次数
- 失败 chunk 数
- 失败类别分布
- 自动瓶颈判断

目标是在 UI 上直接回答：

- 是模型慢？
- 是服务器处理慢？
- 是失败重试拖慢？
- 是后端批量编排不合理？

## 第二阶段：LLM 吞吐和稳定性优化

建议按数据结果逐步实施：

- 动态批大小：稳定时增大批，失败率升高时自动减小批。
- 动态并发：模型稳定时提高并发，错误率升高时自动降并发。
- 慢请求隔离：超过阈值的批次拆小或放入慢段队列。
- fallback 模型：主模型抖动时切换备用模型。
- 输出格式保护：继续强化 JSON 解析和 HTML 标签校验。
- 单段补译异步化：少数失败段落不要阻塞主流程。

## 第三阶段：后端编排优化

- 断点续跑：重启翻译时只处理未完成/失败段落。
- 章节级落库：每章完成后立即持久化，不因末尾失败浪费已完成工作。
- 失败段落独立队列：主流程完成后，失败段落进入补译队列。
- 缓存优先：重复重启时优先复用成功 chunk。
- 慢段标记：反复失败或超时段落在 UI 上可见。
- 任务恢复：支持继续、重启、只重试失败段落。

## 第四阶段：服务器扩容判断

只有当归因数据显示服务器处理占比高时再扩容。

建议规则：

- CPU 高且卡在解包/清洗/打包：升级 CPU。
- 内存不足或 swap：升级内存。
- 多任务排队严重：增加 worker 或扩机器。
- 单本书耗时主要在 LLM：优先优化模型路由、并发、批量和重试策略。

## 当前优先级

1. 上线翻译耗时归因面板。
2. 基于真实任务数据判断主瓶颈。
3. 优先优化模型稳定性和后端重试编排。
4. 最后再决定是否升级服务器。
